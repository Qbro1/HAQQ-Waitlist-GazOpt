# HAQQ-Waitlist-GazOpt

The biggest change happened inside the **Request structure**, which was way too heavy for the blockchain before. In your original version, every single variable was given its own massive storage slot, making a single request take up five slots in total. I reduced the size of the creation and cancellation timestamps to **uint40**, which is more than enough to last for the next 34,000 years, and completely removed the explicit ID from the struct. As a result, the author's address, the funds source, and both timestamps are now automatically compressed by the compiler into **one single 32-byte storage slot**, while the amount takes up the second one. The structure shrunk from five slots to just two, drastically cutting down on storage costs since writing to slots (SSTORE) is the most expensive operation on the EVM.

The second massive optimization was completely removing the redundant mappings (requestIdToIndex, requestExists) and the tracking counter (_nextRequestId). Since your requests are strictly sequential and are never deleted from the array, we don't need extra tracking tables to check if a request exists or to find its index. The request ID is now simply its own index number within the array, and checking if it exists is just a quick boundary check comparing that ID to the total length of the array. This single adjustment completely eliminates the need to make extra, expensive database writes whenever a new request is submitted.
## Global State and Math Refactoring
On top of that, we rearranged the global state variables so that the contract's **currentState** and the **backendSigner** address now share a single storage slot instead of using two separate ones. I also wrapped the counter increments and user nonces inside **unchecked blocks**. Since these numbers increase sequentially by one, they can never realistically overflow, so bypassing Solidity's default overflow checks strips away unnecessary hidden opcodes and saves gas during transaction execution.
Finally, the verification logic for the owner and manager roles was moved into a single internal helper function. This reduced the overall bytecode size of the contract, making its deployment significantly cheaper. The entire contract now runs much faster, looks much cleaner, and will save your users a massive amount of gas on every single transaction.
